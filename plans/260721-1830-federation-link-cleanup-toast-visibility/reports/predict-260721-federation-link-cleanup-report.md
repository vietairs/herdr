# Predict: federation link-close cleanup + toast visibility

## Predicted failure modes (ranked)

1. **Stale cleanup nukes a fresh remount (generation race).** `end_federation_mount` (`src/app/state.rs:1539-1544`) keys purely on `HostKey`, with no `mount_generation` check. `HostKey` (used by `remote_mirrors` and by `worktree_space.key = "federation:{host_key}"`, `src/app/creation.rs:617`) has no generation component either. Sequence: old link ends -> drive task's cleanup event is in flight -> user (or auto-retry) remounts the same host before the event is processed -> `begin_federation_mount` succeeds, new workspaces materialize -> the stale cleanup event now arrives and calls `end_federation_mount(host_key)` + removes-by-`worktree_space.key`, deleting the **new** mount's registry entry and workspaces. `drive_mount_channel` already fences inbound messages by `generation` (`handle_federation_mount_ready`, `workspaces.rs:200`) — the new cleanup event must carry the same `generation` and the handler must verify `self.state.remote_mirrors.get(host_key).map(|m| m.mount().mount_generation) == Some(expected)` before tearing anything down. This is the highest-risk gap; without it the fix trades "stuck forever" for "remount roulette."

2. **Terminal/System toast delivery still silent in server mode even after touching only `workspaces.rs`.** `handle_federation_mount_failed` (`workspaces.rs:228-244`) returns immediately after an early match in `App::handle_internal_event` (`src/app/api.rs:152-156`), so it never reaches the shared tail (`api.rs:305-323`) that calls `crate::terminal_notify::show_notification` / `crate::platform::show_desktop_notification` under `self.local_terminal_notifications`. That tail is what every other Terminal/System-delivered notification goes through. Separately, in server/headless mode, `handle_internal_event_with_forwarding` (`src/server/headless.rs:1982-2255`) only forwards Terminal/System notices to the connected TUI client for match arms it explicitly handles (`StateChanged`, `UpdateReady`, `PaneDied`); `FederationMountFailed` falls into the catch-all `_ => { self.app.handle_internal_event(ev); true }` (`headless.rs:2251-2254`), which never calls `send_flat_toast_to_foreground_client`/`send_notify_to_foreground_client`. **Fixing only `workspaces.rs` will not surface Terminal/System toasts to a real TUI client attached to a server daemon** — a second, dedicated arm mirroring the `UpdateReady` arm (`headless.rs:2182-2219`) is required in `headless.rs` too.

3. **Terminal-runtime shutdown never drains on the AppEvent path.** `close_selected_workspace` (`src/app/actions.rs:1496-1550`) queues detached terminals into `AppState.terminal_runtime_shutdowns` via `remove_unattached_terminal_ids`, but only *callers* invoke `App::shutdown_detached_terminal_runtimes()` (`src/app/runtime.rs:42-49`) afterward — e.g. `handle_workspace_close` does so explicitly (`workspaces.rs:483`), and `drain_api_requests` does so per API message (`runtime.rs:55`). Neither the monolithic run loop's `LoopEvent::Internal` arm (`src/app/mod.rs:1146-1149`) nor headless's catch-all (`headless.rs:2251-2254`) call it. A new `FederationMountEnded`-style handler that calls the workspace-removal path but forgets this explicit call leaks PTY/OS resources for every removed remote pane until an unrelated API request happens to drain the queue.

4. **Restart-duplication (3->5) is corroborated by a second, independent bug: persistence doesn't know about federation.** `grep` for `federation`/`host_key`/`"r:"` in `src/persist/*.rs` returns nothing. `capture_session_save_job` -> `crate::persist::capture` (`src/persist/snapshot.rs:250-275`) snapshots **every** workspace in `AppState.workspaces`, including federation-materialized ones (no filter on `worktree_space.key` prefix or `Workspace::id`'s `r:` prefix). So even after fix #1 is correct, if a session save fires while a federation mount is live, those workspaces are persisted; on the next server restart they are restored as ordinary local workspaces (their remote-backed panes obviously can't actually run), and a subsequent remount adds fresh ones on top -> duplication. This explains the observed 3->5 symptom without even needing the "no cleanup on link-close" bug, and is out of scope for the two approved fixes but should be flagged to the user as a related follow-up, not assumed fixed by this change.

## Risky assumptions in the proposed shape

- **host_key -> workspace mapping is recoverable, but coarse.** Materialized workspaces are tagged via `workspace.worktree_space = Some(WorktreeSpaceMembership { key: format!("federation:{}", host_key), is_linked_worktree: false, .. })` (`creation.rs:616-622`). This is real and query-able, but `is_linked_worktree: false` plus `key` grouping is *exactly* the shape `close_selected_workspace` already uses for linked-worktree bulk-close (see next point) — reuse is possible but must be generation-fenced per finding 1.
- **A safe bulk-removal API already exists and is reusable, with a side effect worth knowing.** `AppState::close_selected_workspace` (`actions.rs:1496-1550`) does NOT just close one workspace: if the selected workspace's `worktree_space` is non-linked and >=2 workspaces share the same `key`, it closes **all of them** in one call (`close_indices` computed at `actions.rs:1503-1520`). Because every federated workspace from the same host shares `worktree_space.key = "federation:<host_key>"` with `is_linked_worktree: false`, pointing `state.selected` at any one of that host's workspaces and calling `close_selected_workspace()` already removes the whole host's workspace set, reassigns `active`/`selected` safely, clears `selection`, and queues terminal shutdowns — this is very likely reusable almost as-is instead of writing new ad-hoc removal logic.
- **Event emission gap, not just an internal-state gap.** `handle_workspace_close` emits one `EventEnvelope { WorkspaceClosed }` for the single requested workspace id (`workspaces.rs:484-490`) even when `close_selected_workspace` silently removes siblings too — that's a pre-existing gap for worktree-grouped closes. If link-close cleanup reuses `close_selected_workspace`, connected API/CLI clients need a `WorkspaceClosed` (or equivalent) event **per removed workspace**, not zero events — per this repo's runtime/client boundary guardrail (CLAUDE.md), a shared runtime fact (workspaces disappearing) must be exposed through the JSON API/event path, not silently mutated server-side only.
- **Race between `LinkClosed` cleanup and an in-flight `mount_remote` for the same host** is the same issue as finding 1, restated: `double_attach_conflict` (`state.rs:1556-1561`) checks `remote_mirrors` membership only, so as long as cleanup removes the entry before a queued remount's dial finishes, or a generation check gates the cleanup, this stays safe; without it, ordering is not guaranteed by anything in the current event-channel (`mpsc::channel`, not ordered-per-host).

## Smallest-safe-change recommendation per fix

**Fix 1 (link-close cleanup):** Add `AppEvent::FederationMountEnded { host_key: HostKey, generation: u64 }` (or reuse `target`+carry `mirror.mount().mount_generation` from the ready payload through the drive task), sent from the drive task's teardown in `handle_federation_mount_ready` (`workspaces.rs:206-221`) instead of only logging. New handler: verify `self.state.remote_mirrors.get(&host_key).map(|m| m.mount().mount_generation) == Some(generation)` (skip silently if not — stale/superseded), else call `self.state.end_federation_mount(&host_key)`, find one workspace index with `worktree_space.key == format!("federation:{host_key}")`, set `state.selected` to it, call `self.state.close_selected_workspace()` (reuses the existing bulk-close-by-key behavior), then `self.remove_plugin_pane_records(...)` + `self.shutdown_detached_terminal_runtimes()` mirroring `handle_workspace_close` exactly, and emit a `WorkspaceClosed` event per removed workspace id for client parity.

**Fix 2 (toast visibility):** In `workspaces.rs::handle_federation_mount_failed`, branch on all four `ToastDelivery` variants the way `handle_notification_show` does (`api.rs:1110-1144`): keep the `Herdr` branch as-is; add `Terminal`/`System` branches gated by `self.local_terminal_notifications` calling `crate::terminal_notify::show_notification` / `crate::platform::show_desktop_notification` (matches the shared-tail pattern at `api.rs:305-323`). Additionally add a dedicated `AppEvent::FederationMountFailed` arm in `headless.rs::handle_internal_event_with_forwarding`, modeled on the `UpdateReady` arm (`headless.rs:2182-2219`), to forward Terminal/System notices to the foreground client via `send_flat_toast_to_foreground_client`/`toast_notify_kind`. Skipping the `headless.rs` half leaves server-mode Terminal/System delivery silently broken.

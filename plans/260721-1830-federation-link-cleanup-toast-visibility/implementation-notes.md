## Code-review fixes (260721, all 5 findings)

- What: while reordering the drive task's teardown to run before the `FederationMountEnded` send (finding 4), fixed a pre-existing bug in `federation_mount_ended_wiring_link_closed_reaches_event_channel`'s polling loop: its `match` had `_ => break` catching both `Ok(None)` (channel closed) and `Err(Elapsed)` (a single 200ms poll timing out), so the loop only ever got one 200ms window before giving up.
- Why: the event is now sent only after teardown (drop `out_tx`, bounded-await `writer_handle` up to 2s, drop `tunnel_guard`) instead of immediately, so it can legitimately take longer than one 200ms poll to arrive; the old loop deterministically failed once the send was no longer near-instant. Confirmed the bug was pre-existing (not caused by the reorder) by isolating the test before vs. after: it consistently failed post-reorder with the old loop body and consistently passed once `Err(_) => continue` replaced `_ => break` (kept `Ok(None) => break` as the only real "stop early" signal).
- Evidence: `cargo test --bin herdr federation_mount_ended_wiring_link_closed_reaches_event_channel -- --test-threads=1` failed 5/5 runs before the loop fix, passed 5/5 after (some runs took ~2.24s, consistent with hitting the bounded 2s writer-handle timeout in the new teardown-before-send ordering). Full broad suite (`cargo test --bin herdr -- --test-threads=4`) is 2696 passed, 0 failed after this fix.
- Reversibility: test-only change, fully reversible; no production-code behavior depends on it.

## AgentStatus relay wiring (260722, follow-up to debug-260721-2357-remote-agent-detection-not-in-sidebar-report.md)

- What: routed `FederationMessage::AgentStatus` frames in `drive_mount_channel` (`src/remote/federation/client.rs`) into a new `TerminalChannelRouter::route_agent_status`, keyed by the wire's raw `AgentStatusMessage::terminal_id` — the same raw-id key space `output_senders`/`open_terminal` already use — instead of resolving through the mirror's namespaced `pane_id` lookup.
- Why: the confirmed key-mismatch risk in the debug report (mirror pane lookup keys on namespaced `terminal_id`, but the runtime's registered sender needed a stable local key) is avoided entirely by using the wire's raw id for routing, since that raw id is identical to the one `build_remote_pane`/`open_terminal` already register terminal output senders under. `RemoteMirror::apply_agent_status` is still called first (unconditionally) purely for its own bookkeeping side effect (mirrored `PaneInfo.agent_status` + `PaneUpdated` event for any other mirror-reading consumer); its `ReducerAction` return value is not used to gate the router forward, so a mirror-side `Ignored` (e.g. value unchanged, or pane not yet hydrated in the mirror) never blocks the pane's own detection loop from receiving the status.
- Evidence: new test `drive_mount_channel_relays_agent_status_to_the_registered_pane_sink` (`src/remote/federation/client.rs`) asserts an `AgentStatus::Working` frame reaches a registered `mpsc::Receiver<AgentStatus>` end to end through `drive_mount_channel`. `cargo test --bin herdr federation` (118 passed) and `cargo test --bin herdr pane::` (274 passed) both green after the change; `#[allow(dead_code)]` removed from `PaneRuntime::relayed_agent_status_sender` (`src/pane.rs`) with a clean `cargo check`, confirming a real production call site now exists (`src/app/creation.rs::build_remote_pane`).
- Reversibility: fully reversible — the new router fields/methods and the `build_remote_pane` registration call are additive; reverting means restoring the old `FederationMessage::AgentStatus(_) => continue` arm and re-adding the `#[allow(dead_code)]`.

## Remote federated pane render corruption fix (260722, resize replay-recovery gate)

- What: added `PaneRuntimeIo::is_remote()` (`src/pane.rs`) returning true only for `PaneRuntimeIo::Remote(_)`, threaded a new `is_remote_backed: bool` parameter through `PaneRuntime::resize` -> `PaneTerminal::resize` -> `GhosttyPaneTerminal::resize` (`src/pane.rs`, `src/pane/terminal.rs`), and gated the resize-recovery ANSI replay block (`GhosttyPaneTerminal::resize`, `src/pane/terminal.rs:~1365-1374`) off entirely when `is_remote_backed` is true (`replay_ansi` short-circuits to `None`).
- Why: per debug report `plans/260721-2353-federation-agents-sidebar-remote-detection/reports/debug-260722-0001-remote-pane-render-corruption-report.md`, the replay-recovery heuristic assumes a near-synchronous same-machine SIGWINCH repaint (true for local PTY panes) but a remote-backed pane's authoritative repaint is an async round trip over the federation link (`pane_source.rs` RT-F10 comment); rapid drag-resize repeatedly fired the replay before the real remote repaint returned, stacking stale content under the eventual authoritative bytes.
- Evidence: `cargo build --bin herdr` clean; `cargo test --bin herdr pane:: -- --test-threads=4` 275 passed, 0 failed, including new test `resize_recovery_skips_replay_for_remote_backed_pane` (`src/pane/terminal.rs`) plus the 3 pre-existing local-behavior resize-recovery tests (all now pass `is_remote_backed = false` explicitly, unchanged assertions); `cargo clippy --bin herdr` shows 0 new errors.
- Reversibility: fully reversible; the new parameter and `is_remote()` helper are additive plumbing, and the gate is a single `!is_remote_backed &&` condition added to the existing `replay_ansi` branch — local pane behavior (all existing tests pass `false`) is byte-for-byte unchanged.

## Remote split protocol scaffolding + misfile guard (260722, fix for debug-260722-0002-remote-split-spawns-local-pane-report.md)

- What: added `FederationMessage::SplitPaneRequest`/`SplitPaneResponse` (+ `SplitDirection`) to `src/remote/federation/protocol/mod.rs` (bumped `FEDERATION_PROTOCOL_VERSION` 2 -> 3, added to the codec's exhaustive round-trip fixture), added `FederationHost::split_pane` to the trait in `src/remote/federation/serve.rs` with a request/response handler in `handle_inbound`, implemented it on the test-only `FixtureHost` (`src/remote/federation/loopback.rs`, mints a sibling terminal id or fails for an unknown target), and added a `classify(&public_workspace_id)`-gated refusal in `handle_pane_split` (`src/app/api/panes.rs`) that returns `remote_split_unsupported` instead of falling through to the local-spawn path for any `r:`-namespaced (federation-mounted) workspace.
- Why (scope decision): the debug report's fix shape needs 4 pieces — (1) remote-mount detection in `handle_pane_split`, (2) a new protocol request/response pair, (3) a serve-side handler performing the real split, (4) client-side routing that awaits the response and materializes the new pane via the existing `build_remote_pane`/`spawn_remote` path. This fix delivers (1) and (2) fully, and (3) only for the test-only `serve.rs`/`loopback.rs` path (proven by 2 new loopback tests) — NOT for production. Production's actual serve-side dispatch lives in `src/server/federation_accept.rs`, which duplicates `serve.rs`'s `handle_inbound` logic independently (confirmed by grep: `FederationHost` has exactly one production-shaped implementor, and it is not `federation_accept.rs` — that file hand-rolls its own terminal-channel routing rather than implementing the trait) and was **not in this task's owned-files list** (`src/remote/federation/*`, not `src/server/*`). (4) requires a synchronous JSON-API handler (`handle_pane_split`) to await an async round trip over a live mount's `out_tx`/reader, which no existing call site in `App` supports today (the mount's reader loop is `drive_mount_channel`, a single always-running consumer with no per-request correlation table) — building that bridge safely spans `App`'s mount-registry state (not an owned file) and is a materially larger change than "wire an existing pattern." Per the assignment's own precedent for `src/pane.rs` ("report as a follow-up instead of touching an unowned file"), the same treatment applies here to `src/server/federation_accept.rs` and to the `App`-level request/response bridge.
- Given that gap, the **minimal, non-regressive fix actually shipped** replaces the dangerous silent misfile (a local PTY spawned and stamped with a remote-namespaced public id, becoming visually and structurally indistinguishable from a real remote pane) with an explicit, typed refusal (`remote_split_unsupported`) — the split simply does nothing and the user gets a clear error instead of a corrupted-looking remote workspace.
- Evidence: `cargo test --bin herdr federation -- --test-threads=4` 121 passed (2 new: `split_pane_request_for_a_known_pane_yields_a_created_response`, `split_pane_request_for_an_unknown_pane_yields_a_failed_response`, plus 1 new protocol roundtrip test and the codec's exhaustive-variant test extended); `cargo test --bin herdr -- --test-threads=4` full suite 2702 passed, 0 failed, including new regression `pane_split_in_a_federated_workspace_is_refused_not_misfiled_locally` (asserts pane count is unchanged after a refused remote split); `cargo build --bin herdr` and `cargo clippy --bin herdr --no-deps` show only 2 pre-existing unrelated dead-code warnings (`map_out`, `Capability::CLIPBOARD`), no new warnings/errors.
- Reversibility: fully reversible/extensible — the protocol variants and serve-side trait method are additive (an older peer without them simply cannot decode them, gated by the version bump); the `handle_pane_split` refusal is a single early-return guard that can be replaced by a real remote-dispatch call once the `App`-level request/response bridge and `federation_accept.rs` handler exist, without touching anything else added here.
- Follow-up required (not done, out of ownership/scope): (a) `src/server/federation_accept.rs` needs its own `SplitPaneRequest` handler performing a real split against the live `AppState`-backed workspace (mirroring its existing hand-rolled `TerminalChannelMessage` routing) and a response send; (b) `App` needs a per-mount request-id-keyed response-correlation map (or a oneshot-channel registry) so `handle_pane_split` can enqueue a `SplitPaneRequest` on the mount's `out_tx` and asynchronously await/poll the matching `SplitPaneResponse`, then call `build_remote_pane`-equivalent materialization for `Created`, or surface `Failed`'s reason to the caller; (c) once (a)/(b) land, `handle_pane_split`'s current unconditional refusal should become conditional only on "no live mount found for this workspace", not "workspace is remote" outright.

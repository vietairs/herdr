Ship Gate — Multi-Remote Federated Workspace Launch

2026-07-21 · branch feat/remote-workspace-federation (uncommitted diff, 2286 lines) · plan plans/260721-1403-multi-remote-federated-workspace-launch/

TL;DR

One command now launches the local workspace plus N remote federated workspaces in one TUI: herdr --remote localhost 131.172.248.163 131.172.248.161 --remote-workspace. The federation mount moved server-side — the herdr daemon owns dial/mount via a new JSON API command workspace.mount_remote, and the launching process stays a thin client. --remote accepts multiple targets; localhost means the local workspace (no SSH); mounts live in a HashMap<HostKey, RemoteMirror>; each target dials concurrently and fails independently with a per-host sidebar notice. Tests: 2682/2682, clippy zero new findings, PROTOCOL_VERSION unbumped (JSON-API-only change).

Context

Requested feature: "1 command to connect/start 3 or more herdr workspaces (1 local + 2 remotes)". This reverses the earlier P9.2b decision (federated session renders alone) and lifts the S10.1 single-mount restriction. The original in-process plan was BLOCKED (the local launch path spawns the server as a separate OS process — no in-process AppState); the user chose server-side mounts, which also matches the project's server-owned-runtime guardrail.

Intuition

The CLI parses N --remote targets, filters out localhost, ensures a local server via the existing autodetect machinery, sends one workspace.mount_remote request listing the SSH targets, and attaches as a normal thin client. Inside the daemon, the handler spawns one fire-and-forget tokio task per target (SSH provision + dial + mount, ~25s budget each, concurrent). Results come back into App::run's tick through the existing AppEvent channel (worktree-add precedent): success materializes the mount and starts the drive task; failure raises a per-host toast and leaves everything else running.

Blast Radius

  • Classic single-target --remote (no --remote-workspace) routes through the untouched run_remote/run_federated_session path; explicit acceptance test guards this.
  • --remote-keybindings server combined with --remote-workspace now fails loudly at launch instead of being silently ignored — the federation wire carries raw resolved bytes with no keybindings handshake, so honoring it server-side would need new wire semantics. This is the one user-visible behavior gate.
  • Windows: multi-value parse parity mirrored in src/remote.rs but not compiler-verified locally (no cross-compile). Remote mode still errors on Windows as before.
  • No wire-protocol change; attached clients see mounts through the normal render/event push. No suspicious embedded content found in the diff.

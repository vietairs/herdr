1. TL;DR

Pasting an image while a mounted remote-workspace pane is focused now writes the file on the remote host and pastes back a path that resolves there, instead of writing locally and pasting a path the remote shell cannot see. A new file_staging federation capability carries the image bytes over the existing mount link as base64 on a dedicated 24 MiB channel; the remote stages it under a collision-proof federation-clipboard- prefix in the per-user staging dir; the returned path is re-validated at the injection boundary before it reaches the pane's PTY. No protocol version bump. Unix-only on the serving side, compile-gated so Windows builds are unaffected.

3. Intuition

  • Capture. ctrl+v on a focused mounted-remote pane hits a pure three-branch decision function (Capture / Unsupported / FallThrough). Only the first two consume the key, so local panes and every non-terminal mode keep byte-identical behaviour. A held key emits Repeat events, which are consumed but do not re-read the clipboard — one press, one capture.
  • Send. The client checks the negotiated file_staging capability at a single choke point and returns an error rather than sending to a peer that never agreed. Payload is capped at 16 MiB before it hits the wire, and at most two stages per mount are in flight.
  • Stage. The remote's connection reader thread only routes: it takes an RAII permit and hands the job to a dedicated staging thread, so a slow filesystem never stalls the panes. The filename runs an eight-step ordered contract that rejects rather than repairs, and every rejection is decided before create_new, so a failed stage leaves nothing on disk.
  • Return. The response is matched to its request by id, origin, and a locally minted MountConnectionEpoch — checked in that order, before the pending entry is claimed — so a stale answer after a fast remount cannot evict a live one. The path is then re-validated (absolute, no .., no control bytes, no shell metacharacters, must carry the staging prefix, ≤4096 bytes) and only then pasted.

4. Blast radius

  • Deliberate cross-feature side effect: FederationMountEnded now carries the mount epoch, so a stale end-notice can no longer tear down a live remount for any consumer of that event — split-pane included. Called out in the plan's Deferred section; an improvement, but it changes behaviour outside this feature.
  • Head-of-line blocking is accepted, not fixed (assumption A1). A multi-MiB paste freezes every pane on that mount for the duration of the transfer. A loopback test documents the stall and asserts only that delivery resumes.
  • Windows: all new staging code is #[cfg(unix)]; federation_accept.rs, serve.rs and app/input/mod.rs still compile on Windows. Cross-compile shows exactly the 10 pre-existing errors at the fork base — none naming a symbol this feature adds.
  • Threat model (from the stage-7 review, and it bounds everything): the returned path is pasted into the remote pane's PTY, so a hostile remote injecting metacharacters is attacking its own shell. What it actually gains is control of what the user sees rendered and what the agent in that pane consumes. Direction B is likewise defence-in-depth: the federation listener is a unix socket in the host user's runtime dir, so a "hostile mounting client" is already authenticated as that user.
  • No suspicious embedded content was found in the diff, commit messages, or notes — nothing attempting to alter this gate's behaviour. No secrets or credentials appear in the changed regions.

Pasting a file into a pane that lives on another machine

Three ways to close the gap that currently forces you out of herdr and into cmux ssh appn-ltu-vm-105. Pick one before any code is written.

Why it's broken

Paste already works — it just works on the wrong machine. When you paste an image, the local server writes the bytes to a temp file and types that local path into the pane (server/clipboard_image.rs:13). For a pane on a mounted remote workspace the PTY is running on the far side of the SSH link, so the agent receives a path to a file that does not exist on its filesystem. cmux ssh works precisely because there the file and the agent share a host.

What scouting turned up

The rails for this were already laid, then left dormant. ClipboardMessage and FederationMessage::Clipboard already ship in protocol v3 with a 16 MiB channel cap. send_local_clipboard_to_remote() exists in client.rs:452 with the comment "no production call site until P8 wires a real paste keybinding to a live mount." The receiving end at serve.rs:441 drops it on the floor: "Clipboard forwarding is deferred (M9)."

So this is finishing P8 + M9, not inventing a protocol. The catch: that message carries no extension, no filename, no target pane, and no reply path — which is exactly what the three proposals disagree about.

The three designs

They differ on one axis above all: who writes into the PTY, and therefore whether you ever find out that a paste failed.

A

Finish the dormant rails

Remote stages the file and pastes into its own pane. Nothing comes back.
no version bump smallest — 350-500 LOC loopback-testable silent on failure 2nd PTY-write path
Wire
Add #[serde(default)] fields to the already-shipped ClipboardMessage — same idiom as AgentStatusMessage.agent. Protocol stays v3.
Injects
Remote server, into terminal_id from the message.
Reversibility
Easy — effectively two files.

Riskiest: there is no acknowledgement of any kind. Mount drops mid-transfer, remote disk full, payload over cap — all of it vanishes with nothing on your screen. That is a design property, not a bug to fix later.

B

Stage remotely, paste locally

Remote only writes the file and hands back its path; the existing local paste code types it.
no version bump failures are visible one PTY-write path in the codebase loopback-testable 500-700 LOC
Wire
Two purpose-built variants gated by a new negotiated Capability — the codebase's own documented mechanism for adding a message without a version bump. Mirrors SplitPaneRequest/Response correlation via request_id.
Injects
Local client, through the paste path that already exists and is already exercised.
Reversibility
Medium — protocol variants, capability, and a pending-request map to unwind.

Riskiest: the pending-request map is new state that has to survive mount teardown cleanly — end_federation_mount() aborts tasks, so an in-flight request must time out rather than hang or panic on a dropped channel.

C

Out-of-band over a second SSH session

Keep bulk bytes off the interactive tunnel entirely; a new CLI subcommand does the staging.
no size ceiling protocol untouched needs a whole new test harness 600-900 LOC new remote entry point
Wire
None — this is off the federation protocol. SSH already runs ControlMaster=auto, so a second session is nearly free.
Injects
Either side; the remote would act through its own JSON API, which fits the runtime/client boundary rule in CLAUDE.md better than anything else here.
Reversibility
Hardest — a new subcommand becomes a used interface quickly.

Riskiest: it solves large-file transfer, which nobody asked for, at a cost nobody budgeted. The loopback harness cannot test it at all, so it drags in new test scaffolding and a subprocess-to-mount-teardown lifecycle.

Side by side

A · fire & forget B · correlated C · out-of-band
Protocol version unchanged (v3) unchanged (v3), capability-gated not applicable
Failure is visible no yes — timeout / error yes — exit code
New PTY-write code yes, remote-side none — reuses local paste depends on flavour
Size ceiling 16 MiB channel cap 16 MiB, enforced both ends none until one is added
Fits loopback harness yes yes no
Rough size 350–500 LOC 500–700 LOC 600–900 LOC

Recommendation — B

B costs roughly 150–200 lines more than A and buys the one thing A structurally cannot have: you find out when a paste fails. It also keeps a single PTY-write path in the whole codebase rather than growing a second one on the remote side, and its compat story uses the mechanism the project already documents for this exact situation.

If you want the thinnest possible v1 instead, take A scoped to images only — with the understanding that silent failure is an accepted gap, not an oversight. C is not recommended now; revisit it only if "arbitrary file" turns out to mean hundred-megabyte build artifacts.

Two things worth correcting

  • Image and arbitrary-file are two features, not one. Local capture is image-only today — read_image_file_from_terminal_drop(). All three proposals generalise the wire and staging side; none of them size the work to capture a non-image file in the first place.
  • A live bug sits in the staging path regardless of which you pick. sanitize_extension() silently rewrites any unrecognised extension to .png. Fine when everything is an image; actively wrong the moment you paste report.pdf.

What the plan does next

Open questions

These shape the plan. I'll assume sensible defaults if you'd rather not answer all of them.

  1. Is arbitrary-file really v1? Or is image-paste-over-federation the actual need, with generic files as a fast-follow? This decides whether capture generalisation is in scope.
  2. Filename preservation. Should the agent see screenshot-of-error.png, or is today's opaque client-3-clipboard-1753...png fine?
  3. Anything above 16 MiB? Only matters if the answer changes C's standing.
  4. Remote staging directory lifetime — the same 24-hour sweep as local, or cleared when the mount goes away?

## 1. Step-by-step: what happens on `herdr --remote ssh user@ip`

Entrypoint: `main()` in `src/main.rs:436` → `remote::extract_remote_args(&args)` (`src/remote/unix.rs:61`, re-exported via `src/remote.rs:5`) parses `--remote <target>` out of argv into a `RemoteLaunch{target, keybindings, live_handoff}`. Guard at `main.rs:453-464`: `--remote` may only combine with the default launch command (no other subcommand/flags besides `--session`/help/version).

`main.rs:683` calls `remote::run_remote(remote_launch)` → `src/remote/unix.rs:155` `run_remote()`, which does, in order:

1. `session::active_name()` — resolve session name (default or `--session`).
2. `local_forward_socket_path(target, session_name)` (`unix.rs:1954`) — pick a local Unix socket path in tmpdir (with hashing fallback for the 103-byte `sun_path` limit).
3. `reattach_command(...)` (`unix.rs:1636`) — builds the shell command string used later for "reattach" (re-running `herdr --remote <target> ...`) — passed to the client process as `HERDR_REATTACH_COMMAND` env var (used by the client's reattach-on-disconnect UI, not shown in the excerpt but present as `REATTACH_COMMAND_ENV_VAR`).
4. `RemoteSsh::new(target, manage_ssh_config)` — wraps ssh invocation, optionally through a herdr-managed ssh config (`write_managed_ssh_config`, `unix.rs:1813`) that `Include`s the user's real `~/.ssh/config` first and adds `ServerAlive*` keepalives + a control-socket path for connection multiplexing.
5. `prepare_remote_herdr(&remote_ssh, live_handoff)` (`unix.rs:672`) — **install/ensure step**: detects remote OS/arch (`detect_remote_platform`, `unix.rs:738`, via `uname`), searches for an existing compatible `herdr` binary on the remote (`remote_binary_candidates` → known paths + `$PATH` discovery, `unix.rs:757-903`), and if none found/matching, resolves an install source (`resolve_install_source`, `unix.rs:971`: either seed from the local binary if platform matches, `local_binary_can_seed_remote`, `unix.rs:989`, or download a release asset via `download_release_asset`/manifest fetch, `unix.rs:1446-1568`) and streams it to the remote (`RemoteSsh::install_herdr`, `unix.rs:522`, via `ssh ... cat > tmp && mv` script pattern, `remote_install_stream_command`/`remote_install_commit_script`, `unix.rs:602-614`). If a server is already running remotely, it interactively confirms before overwriting the binary (`confirm_remote_install_with_running_server`, `unix.rs:1094`).
6. `ensure_remote_server_ready(...)` (`unix.rs:1025`) — checks `remote_server_status()` (`unix.rs:1229`, runs `herdr status server` remotely and parses JSON, `parse_remote_server_status_json`, `unix.rs:1267`). If a remote server is already running and compatible (same protocol/version, daemonized), it's left alone — **no explicit "start" call is issued in that case**, because step 8 below (`remote-client-bridge`) auto-spawns a server on demand. If incompatible/stale, it either does live handoff (`live_handoff_remote_server`, `unix.rs:1365`, only when both sides opted into `--handoff`) or stops it (`stop_remote_server`, `unix.rs:1385`, after interactive confirmation via `confirm_remote_server_stop`, `unix.rs:1291`, unless already pre-approved during install).
7. `SshStdioBridge::start(...)` (`unix.rs:1685-1743`) — spawns a background thread that `UnixListener::bind`s the **local** forward socket and, per accepted local connection, calls `bridge_connection()` (`unix.rs:1851`) which spawns `ssh -T <target> "exec <remote herdr path> [--session X] remote-client-bridge"` and pipes the local Unix-socket bytes to/from the ssh child's stdin/stdout verbatim (`copy_flush`, byte-level `io::copy`-style loop).
8. `run_client_process(&local_socket, &reattach_command, keybindings)` (`unix.rs:1923`) — spawns `<current_exe> client` as a **new child process**, inheriting this terminal's stdin/stdout/stderr, with env vars: `HERDR_CLIENT_SOCKET_PATH=<local_socket>`, `HERDR_RENDER_ENCODING=terminal-ansi`, `HERDR_REATTACH_COMMAND=...`, `HERDR_REMOTE_KEYBINDINGS=<local|server>`, and `HERDR_SOCKET_PATH` removed (so it can't accidentally hit the local server's socket).

On the remote side, `remote-client-bridge` is handled at `main.rs:472-474` → `remote::run_remote_client_bridge()` (`unix.rs:194`) → `ensure_remote_server_running()` (`unix.rs:221`): if no server is listening it calls `server::autodetect::spawn_server_daemon()` and waits for the socket; then connects a `UnixStream` to the remote server's own `client_socket_path()` and pipes stdin/stdout of the ssh-invoked `remote-client-bridge` process to/from that socket, again via raw `copy_flush`.

So end-to-end: local `herdr client` process ⇄ local Unix socket ⇄ `SshStdioBridge` thread ⇄ ssh child stdin/stdout ⇄ remote `herdr remote-client-bridge` process ⇄ remote Unix socket ⇄ remote `herdr server`. Four hops, three of which are pure byte pipes; only the two socket endpoints (local client, remote server) speak the actual wire protocol.

## 2. Is there already start/ensure-without-fullscreen-attach logic?

Yes, cleanly separated:
- **Install**: `prepare_remote_herdr()` (`unix.rs:672`) is self-contained and returns just `{remote_herdr, installed_or_replaced, stop_after_install_approved}` — no attach side effects.
- **Ensure running / restart-if-stale**: `ensure_remote_server_ready()` (`unix.rs:1025`) is also self-contained (takes ssh + remote_herdr, returns `io::Result<()>`).
- **Actual "start the server" trigger**: there is no direct "start remote server" RPC called from `run_remote()`. The remote server is lazily spawned as a side effect of the *first* `remote-client-bridge` connection via `ensure_remote_server_running()` → `server::autodetect::spawn_server_daemon()` (`unix.rs:221-238`), which is remote-side code invoked over ssh, not something the local orchestrator calls out-of-band today.

**Reusability**: steps 5+6 (`prepare_remote_herdr` + `ensure_remote_server_ready`) can already run head­less/non-interactively when non-interactive stdin is detected (they degenerate to hard errors instead of prompts, e.g. `unix.rs:1103-1107`) or accept pre-approval flags. They do not touch the terminal or spawn any client. This is a genuine existing seam: "ensure remote herdr is installed and its server is up-to-date and running" is already decoupled from "attach a client to it." The missing piece to "start-without-attach" is just: after `ensure_remote_server_ready()`, issue one lightweight remote call that forces `ensure_remote_server_running()`-equivalent logic (spawn if absent) without going through the bridge/client — e.g. `ssh target "herdr remote-client-bridge"` currently *both* ensures-running *and* opens the data pipe in one step; ensuring-without-piping would need a tiny new remote subcommand (e.g. `herdr server ensure-running`) reusing `server::autodetect::spawn_server_daemon()`, OR simply keep using `remote-client-bridge` as-is and treat its stdio as the data channel regardless of what mounts it (see §5).

## 3. Data channel

One bidirectional stream per attach, **not multiplexed** at the transport layer:

- Local socket (`local_forward_socket_path`) is a **Unix domain socket** the local `herdr client` process connects to as if it were talking directly to a local server (`client_socket_path` env override, `client/mod.rs:1136-1149` via `crate::ipc::connect_local_stream`).
- That socket's bytes are relayed 1:1 (`copy_flush`, `unix.rs:1905`) into `ssh -T <target> "exec <remote> ... remote-client-bridge"`'s stdin/stdout — a single `ssh` child process per local connection, using the **ssh multiplexing control socket** (`ManagedSshOptions.control_path`) so subsequent connections reuse the same underlying TCP connection to avoid repeated auth (`apply_managed_ssh_options`, `unix.rs:634`).
- Remote-side, those same bytes are relayed 1:1 into a **Unix domain socket** connection to the remote server's real client socket (`run_remote_client_bridge`, `unix.rs:194-219`).
- What flows over this pipe is herdr's own **length-prefixed binary wire protocol** (`protocol::write_message`/`read_message`, 4-byte LE length prefix, `MAX_FRAME_SIZE` = 2MB, `MAX_GRAPHICS_FRAME_SIZE` = 32MB for Kitty graphics) carrying `ClientMessage`/`ServerMessage` enum variants (`src/protocol/wire.rs:308,599`) — this is the **same protocol used for local (non-remote) client↔server**, not a special remote-only protocol. It is a single connection, not multiplexed into separate channels for input/output/clipboard/etc. — everything (keystrokes, resize, frame data, clipboard OSC52, notifications, window title) is interleaved as typed messages on the one stream.

## 4. Where is "full-screen" decided? Opaque stream vs structured events?

Two layers of "opaqueness," both hard constraints today:

**a) Wire-level encoding**: `RenderEncoding` (`protocol/wire.rs:39-43`) has exactly two variants:
- `SemanticFrame` (default for local clients, `client/mod.rs:655-660`) → server sends `ServerMessage::Frame(FrameData)` where `FrameData` (`protocol/wire.rs:460`) is `{cells: Vec<CellData>, width, height, cursor, hyperlinks, graphics}` — a **complete rendered cell grid of the server's entire ratatui UI** (i.e. the whole herdr app screen, including its own sidebar/workspace chrome), not a decomposed tree of workspaces/panes/tabs.
- `TerminalAnsi` → server sends `ServerMessage::Terminal(TerminalFrame)`, pre-diffed raw ANSI byte sequences for direct terminal writing.

`run_remote()` forces `HERDR_RENDER_ENCODING=terminal-ansi` (`unix.rs:1935`) on the spawned client, so remote attach **always** negotiates `TerminalAnsi`, the more opaque of the two (raw bytes, no cell structure at all). But even `SemanticFrame` is a **single pre-composed full-screen render**, not per-workspace/per-pane structured events — so switching encodings alone would not unlock a "route into local workspace" model; the server-side rendering pipeline itself produces one full UI frame per tick regardless of encoding, there is no concept of the client requesting "just this pane's content."

**b) Process/terminal level**: `run_client_process()` (`unix.rs:1923`) spawns an entirely separate `herdr client` **child process** that inherits the *outer terminal's* raw stdin/stdout/stderr (`Stdio::inherit()`). That child then calls `setup_terminal()`/`setup_direct_attach_terminal()` (`client/mod.rs:1186-1194`) which enters raw mode / alt-screen on the real terminal for the lifetime of the process. Critically, **this remote client process is completely independent of any local herdr server/session** — `--remote` bypasses the local session/server entirely (guarded at `main.rs:453-464` to be mutually exclusive with other subcommands). There is no "local herdr instance" running concurrently that the remote attach could hand events to; the whole local terminal is handed to this one purpose-built child process until it exits.

So "full-screen" is decided at two points: (1) the wire encoding is forced to the most opaque option, and (2) architecturally, remote attach today never runs inside a local server/session at all — it's a standalone client process taking over the real terminal.

## 5. Feasibility of mounting remote as a local workspace

**Rating: Large** (not a rewrite, but touches both the remote-transport layer and the local server/session core; not a small/medium tweak).

Reasoning:

- **Reusable as-is (small effort)**: `prepare_remote_herdr()`, `ensure_remote_server_ready()`, ssh-config management (`write_managed_ssh_config`/`ManagedSshOptions`), and the raw byte-pipe primitives (`SshStdioBridge`, `bridge_connection`, `copy_flush`) are all cleanly factored and free of terminal/UI assumptions. They already separate "get a compatible remote server running" from "attach." This is a genuine existing seam — the install/ensure logic requires ~no change to be invoked from a different caller (e.g. the local server process instead of the CLI entrypoint).
- **Needs new plumbing (medium-to-large)**: Today the local *client* opens the local forward socket and pipes to ssh (`run_remote()` runs entirely in the one-shot `herdr` CLI process, not inside a persistent server). For federation, the **local herdr server** (the persistent daemon backing the local session) would need to own the ssh bridge/lifecycle itself — i.e. move `prepare_remote_herdr`/`ensure_remote_server_ready`/`SshStdioBridge` (or equivalents) into the server process, keyed by workspace, with lifecycle tied to the workspace's existence rather than to a single CLI invocation's process lifetime. That's a meaningfully different control flow than the current "one command, one attach, exit on disconnect" model (`run_client_process` returning when the child exits).
- **Hard blocker (rewrite-adjacent)**: the wire protocol has no concept of "give me just this workspace's panes as structured content I can embed." Both `RenderEncoding` variants produce a **single full-screen rendered frame of the remote app's entire UI** (`FrameData`/`TerminalFrame`). There is no per-pane/per-workspace frame decomposition at the protocol level today. Two realistic paths, both requiring real protocol/rendering work:
  1. **Embed-as-opaque-pane**: keep `TerminalAnsi` full-screen rendering, but instead of the outer terminal being handed to a standalone client process, feed that ANSI stream into a *pane widget* inside the local herdr layout (i.e., the remote mount behaves like a single PTY-backed pane whose "PTY" is actually the remote herdr's whole rendered UI streamed over ssh). This reuses today's rendering/encoding untouched and is the smaller of the two options, but the remote session still can't be decomposed into separate local tabs/panes — it shows up as one nested "screen inside a pane," and remote keybindings/mouse routing need care (double UI chrome: remote's own sidebar rendered inside a local pane).
  2. **True structural federation**: extend the wire protocol so the remote server can expose its workspace/pane tree and per-pane content (not a single composited frame), so the local client's existing workspace model can enumerate/render the remote's panes natively alongside local ones. This requires new `ServerMessage`/`ClientMessage` variants, changes to whatever local in-memory workspace/session model exists (not reviewed in this pass — outside `src/remote*`/`src/client*`), and almost certainly protocol version bump handling (`PROTOCOL_VERSION`, currently 16, already has explicit compat-checking machinery in `remote_server_restart_reason`/`parse_remote_server_status_json` that would need a new case for "structural federation supported").

**Bottom line**: the *install/ensure/transport* pieces in this area are already well-separated and reusable with modest changes (wire the ssh bridge into the server process, add a headless "ensure running" remote call). But "mount as a workspace" in the full structural sense (separate panes/tabs from the remote coexisting individually in the local sidebar) is blocked by the wire protocol only ever sending one composited full-screen frame — that part is a genuinely new protocol capability, not a re-wiring of existing code. The pragmatic middle ground (option 1 above: nest the remote's existing full-screen `TerminalAnsi` stream inside one local pane) is achievable without protocol changes and could plausibly ship as a first iteration, with true per-pane federation as a larger follow-up.

Mounting herdr --remote as a local workspace

Decision doc — Stage 2 brainstorm · herdr fork · 2026-07-13

discovery-only, no code changed 4 scouts synthesized

The problem

Today, herdr --remote ssh user@ip connects out, ensures a compatible herdr server is installed and running on the remote host, then tunnels a local Unix socket through ssh stdio to the remote server's client socket and hands the entire local terminal to a freshly spawned herdr client child process attached to that tunnel (run_remote(), src/remote/unix.rs:155-192). This is session replacement, not federation, for two structural reasons: (1) the wire protocol only ever carries one composited full-screen frame per tick — ServerMessage::Frame(FrameData) / Terminal(TerminalFrame) (src/protocol/wire.rs:39-43,460,599) render the server's entire ratatui UI (sidebar included), never a decomposed per-workspace/per-pane stream — and (2) the remote client is a standalone process (run_client_process, unix.rs:1923) that inherits raw stdin/stdout and enters alt-screen for its own lifetime, with zero awareness of any local server or session (main.rs:453-464 even guards --remote to be mutually exclusive with other subcommands). There is no local AppState running concurrently to hand events to. The user's actual ask — local + remote workspaces coexisting in one sidebar — requires either bypassing this whole-screen swap (feed a remote pane through the existing local pane infrastructure instead) or building a genuinely new protocol capability to decompose a remote server's session into structured, embeddable pieces.

Three approaches

A — Tier‑1 "pane‑tunnel" (small/medium, no protocol change)

Keep the local herdr server as the only AppState in play. A new/changed CLI path (e.g. herdr --remote ssh user@ip without the current full-screen guard, or a new herdr remote-pane) opens a local workspace and spawns one or more ssh-fed panes inside it, reusing the already-decoupled install/ensure machinery. Each remote pane is just a local Pane/TerminalRuntime whose PTY command is ssh -tt user@ip <remote shell or remote-herdr-attach> — the existing spawn_with_portable_pty path accepts any CommandBuilder, zero PTY-layer change needed. Rendering, screen-based agent detection, and cold-snapshot persistence all already operate downstream of process_pty_bytes and are source-agnostic, so they work unmodified. Remote agent status is screen-scrape only (no process-table relay); no warm handoff for remote panes (SCM_RIGHTS fd passing is local-machine-only and architecturally can't cross a network).

Concrete touch-points:
  • Reuse prepare_remote_herdr (src/remote/unix.rs:672) + ensure_remote_server_ready (unix.rs:1025) — already self-contained, no attach side effects, headless-safe.
  • New pane-spawn call using spawn_with_portable_pty (src/pty/backend/unix.rs:12-42) with a CommandBuilder::new("ssh") instead of local shell.
  • on_read closure / process_pty_bytes (src/pane.rs:1722-1755, src/pane/terminal.rs:183,1034) — untouched, feeds the ghostty grid regardless of byte source.
  • Pane::render (src/pane.rs:2493) — untouched, already reads only the grid.
  • Screen-text detection: terminal.detection_text() (src/pane/terminal.rs:1668-1674) + detect_agent_with_osc (src/detect/mod.rs:204-227) — works unmodified. Process-table probe (src/detect/mod.rs:266, src/pane.rs:620-692) does not work for a remote PID — silently degrades or is disabled for remote panes.
  • Workspace::new_with_tab (src/workspace.rs:344-417) — used as-is to create the local workspace container; no id-namespacing needed because these are ordinary local workspace/pane ids (the "remote-ness" lives only in the pane's spawn command).
  • Cold snapshot: PaneSnapshot/launch_argv (src/persist/snapshot.rs:97-108) already generalizes — store the ssh argv, restore respawns it.

B — Tier‑2 "true federation" (large / new subsystem)

The local server becomes a live API client of the remote server: it opens the ssh-tunneled socket, issues session.snapshot (src/api/schema/session.rs, already a fully structured {workspaces, tabs, panes, layouts, agents} description) and events.subscribe, then mirrors the remote's Workspace/Tab/Pane objects live inside its own AppState.workspaces (src/app/state.rs:1323), so the remote workspace renders, resizes, and receives input exactly like a native one. This requires four genuinely new pieces of infrastructure, none of which exist today.

Concrete touch-points / new work:
  • ID remapping layer — generate_workspace_id is a per-process AtomicU64 starting at 1 (src/workspace.rs:73-79,76-79); two servers' w1 collide the instant both appear in one sidebar. Needs prefixing/translation across every *Target in Method/ResponseResult (src/api/schema/*.rs) and wire.rs::ClientMessage::AttachTerminal/ObserveTerminal/ControlTerminal target strings.
  • Multi-source EventHub — src/api/event_hub.rs is a 46-line single in-process Arc<Mutex<Vec<(seq, EventEnvelope)>>>, push()-only from local mutation call sites (e.g. src/app/creation.rs:306); no "ingest external event" entry point exists. Needs a provenance/sequencing redesign or a parallel per-origin hub merged at src/api/subscriptions.rs (825 lines).
  • Remote-backed pane source abstraction — PaneState today is just {attached_terminal_id: TerminalId, seen: bool} (src/pane/state.rs:6-11), a bare 1:1 binding to a locally-spawned TerminalRuntime. No enum PaneSource{Local,Remote} exists anywhere. Needs a new I/O actor reading an SSH channel instead of poll(2) on a raw fd (today's actor, src/pty/actor/unix.rs:66-231, src/pty/fd.rs:127-195, is hard-wired to a local fd).
  • Cross-socket frame/input relay — nothing today proxies ServerMessage/ClientMessage frames between two live socket connections; client_transport.rs (1343 lines) assumes "one socket ↔ this server's own AppState." ClientConnectionMode (src/server/clients.rs:9: App / TerminalAttach / TerminalObserve) has no "proxy" mode.
  • Remote agent-status relay — process-table introspection (src/detect/mod.rs:266, platform-specific /proc/sysctl/Win32) needs to run on the remote host and be shipped back as a substitute return value; no such relay protocol exists.
  • Warm handoff exclusion — SCM_RIGHTS fd passing (src/server/handoff.rs:377-450, src/handoff_runtime.rs:34-39) cannot cross a network at all; remote-backed workspaces must be excluded from local self-update handoff and reconnected after restart instead.
  • Sidebar grouping by host: workspace_list_entries_inner (src/ui/sidebar.rs:318) already groups by worktree_space().key — a parallel "remote origin" grouping key is the one genuinely cheap piece of this tier.

C — Hybrid: ship A now, lay groundwork for B

Ship approach A as the user-visible feature. In parallel, make two small additive changes that don't change current behavior but remove future rework: (1) add an id-origin/namespace field (optional, #[serde(default)]) to workspace/pane id-bearing wire structs and to WorkspaceSnapshot (src/persist/snapshot.rs) so a future federated workspace can be tagged without a breaking schema change; (2) add a session.snapshot ingestion stub — a function shape that can take an external SessionSnapshot and construct local Workspace objects from it — even if nothing calls it yet, so Tier‑2's hardest design question (how does a foreign snapshot become local state) gets settled once, in isolation, instead of under feature-delivery pressure later.

Recommendation

Ship A, defer B

The smallest change that satisfies "remote as a local workspace in one sidebar" is approach A: a local workspace whose pane(s) are spawned via ssh -tt user@ip … reusing prepare_remote_herdr + ensure_remote_server_ready for install/health-check, and the existing spawn_with_portable_pty/CommandBuilder path for the pane itself. Every layer below the pane boundary — terminal emulation, rendering, screen-based agent detection, cold-snapshot persistence — is already byte-source-agnostic by construction (confirmed across all four scouts), so this genuinely is a small/medium change, not a hidden large one.

Explicitly defer: native remote process-table status (needs a remote-side relay protocol that doesn't exist), warm handoff for remote panes (architecturally blocked by SCM_RIGHTS being local-machine-only, not a sequencing problem — this stays deferred under B too, forever, unless the remote server handles its own handoff independently), and true per-pane structural federation (id-namespacing + multi-source EventHub + remote API-client role). Be honest with the user up front: Tier‑2 is not a "later phase" of Tier‑1's code, it's a different subsystem built next to it — expect it to cost multiple times what A costs, concentrated in EventHub, id generation, and a new proxy role for ClientConnectionMode, none of which A's implementation will reduce. If B is ever pursued, do it as its own project with its own plan, not as an "extend A" ticket.

Decide up front whether A changes --remote's default behavior (blast-radius risk to existing full-screen users) or ships as a new opt-in flag/subcommand — the scouts found no code reason it must replace the default, only that today's default happens to be full-screen.

Open questions

  • Does TerminalRuntime (src/terminal/) already abstract over I/O transport (PTY fd vs. arbitrary stream), or is it hard-wired to portable-pty? Not confirmed by any scout — determines whether a "socket-backed runtime" variant (avoiding the ssh-subprocess shortcut in A) is a medium change or requires a new actor from scratch.
  • Remote agent-status relay without a full remote herdr server: is a full remote server always assumed running (as --remote requires today), or could a lighter SSH-exec'd /proc probe suffice for A's "upgrade status past screen-scrape" path?
  • Version-skew handling across local↔remote herdr versions (wire PROTOCOL_VERSION currently 16) — --remote already has compat-checking machinery (remote_server_restart_reason, parse_remote_server_status_json); does approach A need any of it, or does spawning a plain ssh shell pane sidestep protocol compatibility entirely (yes for a bare shell pane, no if A's pane instead embeds a full remote-herdr TUI)?
  • For approach A: should the remote pane be a bare remote shell, or the full remote-herdr TerminalAnsi stream nested inside one local pane (double UI chrome, but preserves the remote's own richer UI)? The scouts flag both as viable but did not pick one — this is a UX call, not an architecture constraint.
  • Does the existing SshStdioBridge/local_forward_socket_path machinery need to move into the persistent local server process (workspace-lifetime-scoped) for A, or can it stay a one-shot CLI-invocation concern per pane spawn? Affects whether reconnect-after-disconnect is trivial or needs new lifecycle code.
